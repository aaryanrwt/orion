# Orion Architecture

This document describes the design and internal architecture of Orion v0.1.0-beta. It is written to be accessible to intermediate Go developers and future contributors who want to learn how Orion works or build upon the networking layers.

---

## High-Level Overview

Orion is a decentralized command execution mesh designed for developers to run commands across their own computers without a centralized coordinator or cloud backend.

```
                      ┌──────────────────────┐
                      │      Orion CLI       │
                      │  (internal/cli/...)  │
                      └──────────┬───────────┘
                                 │
                                 ▼
                      ┌──────────────────────┐
                      │    Engine Interface  │
                      │  (internal/core/...) │
                      └──────────┬───────────┘
                                 │
                 ┌───────────────┴───────────────┐
                 ▼                               ▼
       ┌──────────────────┐            ┌──────────────────┐
       │   Mock Engine    │            │   Real Engine    │
       │ (internal/core)  │            │    (Future)      │
       └──────────────────┘            └──────────────────┘
```

The system is split into three main packages:
1. **`internal/cli`**: The user interface layer, built using Cobra, which handles argument parsing, semantic color output, and interactive spinner/progress rendering.
2. **`internal/ui`**: UI rendering utility containing formatting primitives for grids/tables, spinner animations, and standard diagnostic error boxes.
3. **`internal/core`**: Core configuration models and the orchestrator backend. The execution orchestrator is hidden behind the `Engine` interface.

## Core Interfaces

To facilitate decoupling and testing, Orion isolates networking, pairing, and command execution behind interfaces in [interfaces.go](../internal/core/interfaces.go):

```go
type DeviceManager interface {
    AddDevice(dev Device) error
    GetDevices() ([]Device, error)
}

type CommandRunner interface {
    RunCommand(ctx context.Context, devices []Device, cmd string) <-chan ExecutionLine
}

type Engine interface {
    DeviceManager
    CommandRunner
}
```

* **`DeviceManager`**: Handles saving and retrieving paired devices to/from `config.json`.
* **`CommandRunner`**: Manages execution. In the real implementation, this streams inputs and outputs across secure mTLS TCP connections. In version 0.1.0-beta, it runs via the `mockEngine` to simulate latency and high-fidelity output.

## Configuration and Identity

Orion uses a zero-config model. Identity is generated on the first launch of `orion init`:
* **Device ID**: Generated by combining the local machine's sanitized hostname and a random 4-byte hex suffix (e.g., `laptop-dc11b24a`).
* **Cryptographic Keys**: A mock public/private keypair is generated and saved.
* **Storage Location**: Stored as JSON in standard OS locations retrieved via `os.UserConfigDir()`:
  - **Linux/Unix**: `~/.config/orion/config.json`
  - **macOS**: `~/Library/Application Support/orion/config.json`
  - **Windows**: `%APPDATA%\orion\config.json`

---

## Execution Model

When `orion run <command>` is invoked:
1. **Target Gathering**: The initiator queries `DeviceManager` for all trusted remote nodes and combines them with a representation of the local machine.
2. **Concurreny Orchestration**: A goroutine is spawned for each device.
3. **Real-time Streaming**: Each goroutine establishes a connection to its target device and streams output line-by-line into a shared channel.
4. **Column Alignment**: The CLI gathers these lines and prints them with aligned prefixes (e.g., `[sirius]  Darwin...`), using consistent margins so that logs remain highly readable.
